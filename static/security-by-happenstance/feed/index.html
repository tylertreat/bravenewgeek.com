]]> By: Tyler Treat https://bravenewgeek.com/security-by-happenstance/#comment-25846 Tue, 09 Apr 2019 14:26:43 +0000 https://bravenewgeek.com/security-by-happenstance/#comment-25846 Thanks for the thoughtful comment. A few thoughts:

> The argument is that consuming the credential in Spinnaker is inherently more reliable than consuming the credential in an external service, such as CircleCI. There is an argument for separation of accounts for the build and deploy step, but it relies on the assumption that spinnaker is inherently more secure.

I don’t think Spinnaker is inherently more “secure”, but by nature of it being run inside the cloud environment means it can rely on cloud-managed credentials (e.g. GCP service accounts or AWS IAM roles) as opposed to user-managed keys. Google, for example, rotates the keys for these approximately every 2 weeks. The key isn’t exposed to the user at all. As soon as we have to manage a key ourselves, I think there’s more risk exposure.

> However, from a practical standpoint I do not see it as reasonable that developers are inherently more untrustworthy than whoever is managing spinnaker. Rather, they’re both untrustworthy and there should be no way to recover build credentials from the logs.

Agreed, but I think there’s something to be said about separation of duties. No one person should be able to push a change end-to-end by themselves. And you hit the nail on the head about not being able to recover creds from logs.

> If we have git access we already have access to the production infrastructure by way of modifying the application

Git access in CI should be a non-event since, ideally, secrets are managed separately and injected at deploy time by the CD component. This is where the separation of CI and CD matters IMO.

> Even in the case users can dump the credentials a better solution is short term credentials.

100% agree. Short-term creds is the ideal solution.

> Additionally I can understand why running with production credentials in third party infrastructure might not be reasonable (that is, mounting service accounts in CircleCI) but if this was the case an equally valid solution would be to move the entire CI/CD pipeline into the trusted compute zone, mitigating that model.

Yep, running it fully inside the cloud environment is arguably “more secure” due to the point above about managing keys. This is the delicate balance between relying on SaaS (and reducing operational overhead) and security. If relying on “untrusted” SaaS causes us to overcompensate with our security model and supporting machinery (like automated key rotations), it might actually be a net negative in terms of operations overhead.

]]> By: Andrew Howden https://bravenewgeek.com/security-by-happenstance/#comment-25843 Tue, 09 Apr 2019 10:22:52 +0000 https://bravenewgeek.com/security-by-happenstance/#comment-25843 Yo! I really enjoyed this. I understand the nature of the changing advice from NIST and the value of ensuring that we can allocate all actions to a specific identity (build job, for example), but I’m unsure that I can quite make the jump to isolating CI/CD from production infrastructure in its entirety.

Specifically:

> This is why splitting out Continuous Integration (CI) from Continuous Delivery (CD) is important. If, instead, Circle was only responsible for CI and we introduced a separate component for CD, such as Spinnaker, we can solve this problem. Using this approach, now Circle only needs the ability to push an artifact to a Google Cloud Storage bucket or Container Registry. Outside of the service account credentials needed to do this, it doesn’t need to deal with secrets at all. This means there’s no way to dump out secrets in the build because they will be injected later by Spinnaker.

The argument is that consuming the credential in Spinnaker is inherently more reliable than consuming the credential in an external service, such as CircleCI. There is an argument for separation of accounts for the build and deploy step, but it relies on the assumption that spinnaker is inherently more secure.

I can understand why we could consider spinnaker more secure if, for example, developers do not have access to it. In this case anyone with git access cannot dump protected environment variables to the build logs; either there are no logs, or the untrusted group (developers) can not access the logs to implement the change. However, from a practical standpoint I do not see it as reasonable that developers are inherently more untrustworthy than whoever is managing spinnaker. Rather, they’re both untrustworthy and there should be no way to recover build credentials from the logs.

I understand that it’s not super difficult to dump credentials given git access to CI. However, I would wish to point out:

1. If we have git access we already have access to the production infrastructure by way of modifying the application and
2. Even in the case users can dump the credentials a better solution is short term credentials.

Additionally I can understand why running with production credentials in third party infrastructure might not be reasonable (that is, mounting service accounts in CircleCI) but if this was the case an equally valid solution would be to move the entire CI/CD pipeline into the trusted compute zone, mitigating that model.

I appreciate the work you put into the post. It is thought provoking, and I enjoyed it. But I would like to push back a little at the notion that CI and CD must inherently be decoupled.
