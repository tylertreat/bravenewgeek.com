]]> By: darkfader https://bravenewgeek.com/sometimes-kill-9-isnt-enough/#comment-64 Thu, 13 Nov 2014 06:52:29 +0000 https://bravenewgeek.com/sometimes-kill-9-isnt-enough/#comment-64 I like the IPTables approach and used that or simpler ifconfig down’s a lot.

Pulling the power plug when you just need to pull the network cable (for the error scenario you’re testing) is not really great.

Being able to simulate corruption can go a long way, just need to make sure this is tested well before things go to production (it can _then_ be later tested continually) -> like in your sentence “Simulating failure is a necessary element for building reliable distributed systems, but system behavior isn’t black and white, it’s a continuum. ”

Also, the worst scenario isn’t the outage or the partition, but the coming-back-of-still-running services afterwards.
I.e. in a “sunny days only” cluster like hearbeat you could provoke a lot of fun just by bringing back a node in mid-forced-takeover.

Finally, hug your state machines after each test :)
